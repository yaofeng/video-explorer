20 帧预览 — 3 种实现方案对比

选择帧提取和存储策略

A

逐帧 API（懒加载）

做法：后端提供 GET /api/frames/{id}?frame=N，前端按需逐个请求 20 帧。每帧即时抽取即时返回，不存盘。

Pros

  • 零额外磁盘占用
  • 首帧响应最快（只抽 1 帧就显示）

Cons

  • 20 次 HTTP 请求/每次打开浮层
  • 每次打开都要重新抽帧（无缓存）
  • NAS 反复读同一文件
B

批量抽取 + 独立缓存文件 ⭐ 推荐

做法：后端提供 POST /api/frames/{id}/generate 触发一次性批量抽取，20 帧存为 {stem}.frame00.jpg ~ {stem}.frame19.jpg。前端轮询就绪状态，逐帧填入小图条。已抽取过的视频秒开。

Pros

  • 只抽一次，后续打开秒开
  • 浏览器独立缓存每帧（Cache-Control）
  • 和现有 thumbgen 架构一致（懒生成 + 缓存）
  • 右键切换已缓存帧时零延迟

Cons

  • 每视频多 20 个 JPEG 文件（~20×30KB ≈ 600KB）
  • 首次打开需等待抽帧（10~30s）
C

合成雪碧图（Sprite Sheet）

做法：后端将 20 帧拼成一张大图（4×5 或 5×4 网格），前端用 CSS background-position 切换显示区域。

Pros

  • 单次 HTTP 请求拿到全部帧
  • 文件数少（每视频 1 个 sprite）

Cons

  • 大图体积 = 20 帧之和（无压缩优势）
  • 小图条仍需裁切 sprite，CSS 复杂
  • 无法单独缓存/失效某一帧
  • 前端实现复杂度高
